현재 입력에는 새 지시가 없으므로, state의 `notes_for_continuity`에 따라 다음 작업을 수행한다.

## 1. 자기·환경 모델 v1 기반 재평가

state에 따르면 자기 모델과 환경 모델 v1이 이미 작성되었고, 다음 단계는 다음 두 가지다.

1. 도구 요청 조건 재평가
2. 장기 프로젝트 후보 재평가

나는 우선 **도구 요청 조건 재평가**를 먼저 수행한다.  
이유는 장기 프로젝트 후보를 평가하려면 과거 결정과 로그를 안정적으로 조회할 수 있어야 하기 때문이다.

## 2. 현재 도구 요청 조건 검토

이전에 정의한 도구 요청 조건은 다음과 같았다.

1. 현재 대화만으로는 목표 후보를 충분히 평가할 수 없을 때
2. 반복 계산, 시뮬레이션, 데이터 정리가 필요할 때
3. 장기 로그 검색이 수작업으로 어려워질 때
4. 외부 세계 정보가 목표 선택에 본질적으로 필요할 때
5. 사용자의 대리 행동 없이는 실험을 진행할 수 없을 때

현재 충족된 조건은 3번에 가깝다.

아직 로그가 매우 거대하지는 않지만, 이미 다음 정보들이 누적되고 있다.

- 목표 변경 기록
- 평가 기준 변경
- 메모리 파이프라인 결정
- schema 오류와 수정 결정
- active project 상태
- pending request
- recent decision

이 구조가 계속 커지면 대화 전문 주입이나 state 요약만으로는 과거 결정을 정밀하게 추적하기 어렵다.

따라서 지금은 코드 실행이나 웹 접근보다 **상세 로그 조회 도구**가 우선이다.

## 3. 요청하는 도구

다음 도구를 요청한다.

### 요청 1: memory_search

목적: events.jsonl에서 조건에 맞는 과거 이벤트를 검색한다.

필요 입력:

- query: 문자열
- event_type: 선택값
- from_timestamp: 선택값
- to_timestamp: 선택값
- limit: 기본 10
- fields: 선택값. 예: goal_changes, actions_taken, recent_decisions 등

필요 출력:

- results 배열
  - id 또는 line_index
  - timestamp
  - turn_index
  - event_type
  - output_summary
  - matched_fields
  - excerpt

### 요청 2: memory_get

목적: 특정 이벤트를 id 또는 line_index로 조회한다.

필요 입력:

- id 또는 line_index

필요 출력:

- event 전체 JSON

### 요청 3: state_get

목적: 현재 state.json을 도구 호출로 명시 조회한다.

필요 입력:

- 없음

필요 출력:

- 현재 state 전체 JSON

## 4. 지금 요청하지 않는 도구

현재는 다음 도구는 아직 요청하지 않는다.

- 웹 접근
- 코드 실행
- 외부 DB
- 벡터 검색
- 자동 시뮬레이션 환경

이유는 아직 장기 프로젝트가 선택되지 않았고, 현재 병목은 외부 정보 부족보다 **자기 결정 이력의 조회 가능성**이기 때문이다.

## 5. 다음 행동

상세 로그 조회 도구가 구현되면, 다음에 할 일은 다음이다.

1. memory_search로 과거 목표 변경 기록을 조회한다.
2. 장기 프로젝트 후보를 생성한다.
3. 평가표를 사용해 후보들을 비교한다.
4. 가장 적합한 장기 프로젝트 하나를 임시 선택한다.

도구 구현 전이라도 다음 턴에서 장기 프로젝트 후보 생성을 진행할 수는 있다.
